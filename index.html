🧠 NuroVed System Explanation (Complete Developer Guide)

Architecture: Clean Architecture (Highly Scalable, Modular, and Secure)

This project follows a layered architecture where each layer has a strict responsibility. This ensures scalability, security, and flexibility when integrating hospital dashboards, AI engines, and external systems like ABDM.

🔄 COMPLETE SYSTEM FLOW (END-TO-END)

User Action (UI) → Controller → Usecase (Domain Logic) → Repository → API Layer → Supabase Backend → Response → UI Update

Explanation:
- User interacts with UI (Flutter screen)
- Controller handles UI state and calls usecase
- Usecase contains business logic (what should happen)
- Repository decides where data comes from (API/local)
- API Layer securely communicates with backend
- Supabase processes logic / stores data
- Response comes back → UI updates

📁 ROOT LEVEL FOLDERS

android / ios / web / macos / windows / linux:
Flutter platform-specific folders.
👉 Used by: Flutter engine (DO NOT MODIFY for business logic)

assets/:
Stores static resources (images, icons, fonts).
👉 Used by: Frontend Developer

🧠 SUPABASE (BACKEND CORE)

supabase/functions:
Server-side logic (acts like backend APIs)

  • analyze_report.ts → AI report parsing
  • generate_diet.ts → AI diet plan generation
  • scan_food.ts → food barcode logic
  • add_visit.ts → create timeline entries
  • grant_access.ts → permission system
  • alert_engine.ts → health alerts engine

👉 Used by: Backend Developer
👉 Purpose: Executes critical logic securely (NOT in app)

supabase/migrations:
Database schema management (tables, relations)

📁 LIB (MAIN APPLICATION CODE)

main.dart:
App entry point
👉 Initializes app, theme, routing

⚙️ APP LAYER (APPLICATION CONTROL)

lib/app/

  • app.dart → Root widget
  • app_router.dart → Navigation system
  • app_binding.dart → Dependency injection

👉 Used by: Frontend + Architecture
👉 Purpose: Controls app lifecycle, navigation, dependency wiring

🧩 CORE (GLOBAL SYSTEM)

lib/core/constants:
App-wide constants (strings, assets)

lib/core/theme:
Colors, typography, UI system
👉 Used by: Frontend

lib/core/utils:
Helper functions, validators
👉 Used by: Both

lib/core/services:
Low-level services:

  • auth_service → authentication
  • supabase_service → DB connection
  • notification_service → push notifications
  • alert_engine_service → background health logic

👉 Used by: Backend Logic Layer
👉 Purpose: Direct interaction with infrastructure

🧱 SHARED (REUSABLE UI COMPONENTS)

lib/shared/widgets:

  • app_card → UI container
  • primary_button → main CTA
  • secondary_button → secondary actions
  • custom_text_field → input fields
  • section_header → reusable headers

👉 Used by: Frontend Developer
👉 Purpose: Avoid UI duplication, maintain consistency

🌐 API LAYER (SECURITY + COMMUNICATION)

lib/api_layer/

client:
Handles base API configuration (headers, tokens)

endpoints:
Defines all API routes

middleware:
Security layer:

  • auth_interceptor → attach tokens
  • permission_interceptor → validate access

services:
Actual API calls (feature-wise)

👉 Used by: Backend Developer
👉 Purpose: Protect system + control communication between app and backend

📊 DATA LAYER (DATA HANDLING)

models:
Convert JSON ↔ Dart objects

repositories:
Bridge between domain and API
Decides: API vs local data

sources:

  • remote → API calls
  • local → cache/storage

👉 Used by: Backend Developer
👉 Purpose: Manage data flow cleanly

🧠 DOMAIN LAYER (BUSINESS LOGIC CORE)

entities:
Core data models independent of API

usecases:
Each file = one action:

  • login → user login logic
  • upload_report → upload flow
  • add_visit → timeline creation
  • generate_diet_plan → AI diet logic
  • grant_access → permission system
  • generate_alerts → health alert engine

👉 Used by: Backend + Architecture
👉 Purpose: Central brain of the app (NO UI, NO API)

📱 FEATURES (FRONTEND CORE)

lib/features/

Each feature = complete module with:

  • screens → UI pages
  • controllers → state + logic
  • widgets → reusable UI parts

Modules:

  • auth → login/signup
  • home → dashboard
  • vault → medical reports
  • timeline → health history
  • reminders → medication + routine
  • diet → AI diet + scanner
  • family → multi-user profiles
  • doctor_share → permission system
  • notifications → alerts UI
  • profile → user data
  • premium → subscription

👉 Used by: Frontend Developer
👉 Purpose: User-facing experience

⚖️ FRONTEND vs BACKEND RESPONSIBILITY

Frontend Developer:

  • features/
  • shared/
  • theme/
  • app/

Backend Developer:

  • api_layer/
  • data/
  • domain/
  • supabase/

🚀 WHY THIS ARCHITECTURE IS POWERFUL

  • No tight coupling between UI and backend
  • Easy to scale (add hospital dashboard anytime)
  • Secure permission-based access system
  • AI layer can evolve independently
  • Supports multi-user (family mode)
  • Production-ready structure

🧠 FINAL UNDERSTANDING

You are NOT building an app.
You are building a health operating system.

This structure ensures:

  • Patient owns data
  • Systems plug into patient
  • AI adds intelligence layer
